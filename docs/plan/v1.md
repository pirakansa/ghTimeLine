# Desktop Stream App v1 Specification

**Status:** Draft  
**Source of truth:** This document  
**Last updated:** 2026-05-22

## 1. Objective

Build a Rust desktop application inspired by Jasper for Ubuntu and Windows users who need a maintained GitHub activity stream client with better control over unread state and host configuration.

The app uses `egui` so the same codebase can be built for Linux and Windows.

## 2. Problem Statement

The current workflow depends on Jasper, which is no longer shipping Linux builds and does not fully match the desired feature set. The new app should preserve the useful stream-based workflow while adding explicit unread management and GitHub Enterprise Server support.

## 3. Target Users

- Individual developers using Ubuntu as their primary environment
- Developers who also use Windows and want the same desktop workflow
- Users of GitHub Enterprise Server in addition to, or instead of, GitHub.com

## 4. Scope

### In scope for v1

- Stream view based on saved GitHub search queries
- Local database-backed unread management
- Configurable API target for GitHub.com and GitHub Enterprise Server
- Bookmarking stream items
- Archiving stream items
- Personal Access Token (PAT) support as the first supported authentication method
- GitHub Search streams for issues and pull requests
- Internal design that does not block future support for additional authentication methods
- Internal design that does not block future support for additional GitHub search resource types

### Out of scope for v1

- Full parity with Jasper
- Mac-specific packaging and support commitments
- Team/shared state synchronization across devices
- Server-side backend components
- OAuth App and GitHub App login flows in the first release
- Additional GitHub search resource types beyond issues and pull requests in the first release

## 5. Product Principles

- **Cross-platform first:** Ubuntu and Windows must be first-class targets.
- **Local-first state:** unread, bookmark, and archive state must be stored locally.
- **Host flexibility:** the app must support both GitHub.com and GHES hosts.
- **Implementation simplicity:** v1 should prefer the simplest approach that keeps future extension possible.

## 6. Functional Requirements

### FR-1 Host configuration

v1 must support exactly one host configuration.

Each host configuration must include:

- A user-visible name
- A scheme such as `https` or `http`
- A hostname
- A REST API base path
- An indicator of whether the host is GitHub.com or GHES
- Credentials for API access

The combination of scheme, hostname, and REST API base path defines the effective REST API base URL used by the application.

For v1, the GraphQL endpoint should be derived by host kind:

- GitHub.com: `/graphql`
- GHES: `/api/graphql`

The design may later make the GraphQL path configurable if needed, but that is not required in v1.

The implementation should keep the data model and application boundaries extensible enough that multiple hosts can be added in a future version.

### FR-2 Authentication

v1 must support authentication by Personal Access Token.

The specification for v1 assumes:

- PAT is the primary supported authentication method
- The application architecture should allow additional auth providers later
- Tokens are associated with a host configuration

PATs are expected to persist across app restarts in v1.

v1 must store the PAT in the application configuration file as plain text, alongside other tool settings.

This is an intentional v1 trade-off and must be documented clearly in the product behavior and implementation notes.

### FR-3 Saved search queries

The user must be able to create, edit, delete, and list saved search queries.

Each saved query must be associated with:

- A host configuration
- A query string
- A user-visible name

Because v1 supports only one host configuration, the UI does not need cross-host query management behavior.

v1 saved queries must target GitHub Search for issues and pull requests.

The design should not prevent future expansion to additional GitHub search resource types, but those resource types are not required in the first release.

Saved queries must be persisted in the local database rather than the application configuration file.

### FR-4 Stream view

The app must display a stream derived from saved search query results.

v1 stream behavior is defined for issues and pull requests.

The stream must allow the user to:

- View items returned for a saved query
- Distinguish unread and read items
- Mark an item as read or unread from the list
- Bookmark items
- Archive items
- Trigger a manual refresh on demand
- Open the selected item in the system default web browser

The v1 list view must support direct item actions for:

- Mark read
- Mark unread
- Bookmark
- Archive
- Open in external browser

v1 must support filtering the current query view with:

- Open
- Unread
- Bookmarked

The application must support both automatic polling and manual refresh.

v1 automatic polling must default to a 5 minute interval, and the user must be able to change that interval.

The default stream ordering in v1 must be `updated desc`.

v1 must also allow the user to change the stream ordering.

The primary v1 screen must use a two-pane layout:

- Left pane for saved query selection, unread count badges, and query management entry points
- Right pane for the item list of the currently selected query

The left pane must include a library section with global entries for Inbox, Bookmark, and Archived, separate from saved query entries.

For each stream item in the primary list view, v1 must display at least:

- Title
- Repository name
- Item type (issue or pull request)
- Item number
- Updated timestamp
- Unread state
- Author
- Assignees
- Pull request review status when applicable
- Comment count
- Item state (for example open or closed)
- Labels

When the same issue or pull request matches multiple saved queries, library views and other aggregated lists must display it as a single item.

Grouping and ordering rules are defined by the SQLite storage contract in section 8.

### FR-4a Search API polling constraints

The design must account for the behavior of GitHub Search API polling.

In particular:

- Automatic polling must coexist with user-triggered manual refresh
- The default polling interval is 5 minutes
- The polling interval must be user-configurable
- The default ordering is `updated desc`
- User-selectable ordering must be supported in v1
- Query definitions that depend on mutable state such as open/closed, merged, or draft may require special handling
- The application must not assume that polling alone can always infer every transition reliably
- If filters are introduced to compensate for API limitations, the distinction between upstream query and local filter must be explicit in the design

### FR-4aa GitHub API strategy

v1 must not rely on the REST Search API alone for all displayed pull request metadata.

The expected API strategy is:

- Use the GitHub REST Search API as the primary way to discover issue and pull request matches for saved queries
- Use GitHub GraphQL selectively to enrich already-discovered items with additional metadata that is difficult or inefficient to obtain from REST search responses alone
- Merge the enriched metadata into the local database representation before rendering the list

At minimum, the design should assume GraphQL enrichment is needed for pull request-related metadata such as review requests, review state-derived information, draft state, and merged/open distinctions.

### FR-4b Item opening behavior

v1 does not require an in-app detail browser or embedded web view.

When a user chooses to inspect an item in more detail, the application must open the corresponding GitHub or GHES page in the system default web browser.

The design should not prevent a future version from adding an in-app detail view.

### FR-4c Primary layout

v1 must provide a two-pane primary layout.

The left pane is for library entries and saved queries.

The library section must include Inbox, Bookmark, and Archived.

In v1, Inbox is defined as a global view of non-archived items across all saved queries.

Bookmark is defined as a global view of bookmarked items across all saved queries.

Archived is defined as a global view of archived items across all saved queries.

Each saved query entry in the left pane must show its unread count badge.

The right pane is for stream items belonging to the currently selected saved query.

The right pane must provide the primary item filter control for Open, Unread, and Bookmarked views within the selected query.

For v1, the `Open` filter means:

- Issues whose state is open
- Pull requests whose state is open and not merged
- Draft pull requests are included as long as they are open and not merged

v1 does not require a third pane for inline details or auxiliary metadata.

### FR-4d Right pane status presentation

The right pane is primarily a database-backed list view.

Fetched GitHub data must be written to the local database first, and the list UI should render from the local database rather than directly from transient network responses.

For v1:

- Manual refresh, polling, authentication failures, network failures, and API failures must not clear the previously stored list
- When no items exist in the local database for the selected view, the right pane may show a simple zero-items state
- Ongoing communication status should be surfaced separately from the main list, inspired by Jasper's low-disruption status indication rather than replacing the list with a blocking error screen
- Updates to the rendered list are expected to occur when the local database changes and should not require a prolonged loading screen

### FR-5 Unread management

Unread state must be stored in a local database and survive app restarts.

The user must be able to:

- Mark items as read
- Mark items as unread
- See which items are currently unread

Unread state is application-local and does not sync back to GitHub.

For saved query entries in the left pane, unread count badges must exclude archived items.

Archived items may still retain unread state and must be visible as unread within the Archived library view.

### FR-6 Bookmarking

The user must be able to bookmark items locally and later filter or review bookmarked items.

Bookmark state must survive app restarts.

### FR-7 Archiving

The user must be able to archive items locally to remove them from the main working stream without deleting the underlying GitHub resource.

Archive state must survive app restarts.

### FR-8 Configuration file

v1 must persist application settings in a plain text YAML configuration file.

The v1 configuration file path should follow platform conventions:

- Linux: `$XDG_CONFIG_HOME/ghstreamlistner/config.yml` when `XDG_CONFIG_HOME` is set, otherwise `~/.config/ghstreamlistner/config.yml`
- Windows: `%APPDATA%/ghstreamlistner/config.yml`

The configuration file must include at least:

- Host display name
- Host scheme
- Host hostname
- Host REST API base path
- Host kind (GitHub.com or GHES)
- PAT
- Polling interval
- UI settings such as color-related preferences
- Default stream sort selection and persisted user-selected sort state

The configuration file is intended for user-editable application settings and must not be used as the primary store for fetched GitHub items.

The v1 YAML structure must use the following top-level keys:

- `host`
- `auth`
- `ui`
- `refresh`

The corresponding v1 schema is:

```yaml
host:
  name: "Example GHES"
  scheme: "http"
  hostname: "ghe.nanao.co.jp"
  rest_api_base_path: "/api/v3/"
  kind: "ghes" # "github" or "ghes"
auth:
  pat: "ghp_example"
ui:
  theme: "system" # "light", "dark", or "system"
  accent_color: "#4F8CC9"
  default_sort: "updated_desc"
refresh:
  polling_interval_minutes: 5
```

Additional keys may be added later, but v1 must keep the schema shallow and human-editable.

#### FR-8a YAML configuration schema contract

The configuration schema version is implicit in v1. The v1 application must reject unknown enum values but may ignore unknown object keys so a future version can add optional settings without breaking older data unnecessarily.

Required keys:

| Path | Type | Required | Allowed values / constraints | Default when creating a new config |
| --- | --- | --- | --- | --- |
| `host.name` | string | yes | Non-empty after trimming whitespace | `GitHub.com` |
| `host.scheme` | string enum | yes | `https` or `http` | `https` |
| `host.hostname` | string | yes | Non-empty hostname without path; for GitHub.com this is `api.github.com` | `api.github.com` |
| `host.rest_api_base_path` | string | yes | Absolute path beginning with `/`; normalized to end with `/` | `/` |
| `host.kind` | string enum | yes | `github` or `ghes` | `github` |
| `auth.pat` | string | yes | Non-empty; stored as plain text in v1 | none |
| `ui.theme` | string enum | yes | `light`, `dark`, or `system` | `system` |
| `ui.accent_color` | string | yes | CSS-style `#RRGGBB` uppercase or lowercase hex | `#4F8CC9` |
| `ui.default_sort` | string enum | yes | `updated_desc`, `updated_asc`, `created_desc`, `created_asc`, `comments_desc`, `comments_asc` | `updated_desc` |
| `refresh.polling_interval_minutes` | integer | yes | `1` through `1440` inclusive | `5` |

Validation and normalization rules:

- `host.scheme`, `host.kind`, `ui.theme`, and `ui.default_sort` must be parsed case-sensitively.
- `host.hostname` must not include a URL scheme, path, query, fragment, username, or password.
- `host.rest_api_base_path` must be stored with exactly one leading slash and one trailing slash after normalization.
- GitHub.com configuration must use `host.kind: "github"` and `host.hostname: "api.github.com"`.
- GHES configuration must use `host.kind: "ghes"` and should normally use `host.rest_api_base_path: "/api/v3/"`.
- The effective REST API base URL is `{host.scheme}://{host.hostname}{host.rest_api_base_path}`.
- The effective GraphQL URL is `{host.scheme}://api.github.com/graphql` for `github` and `{host.scheme}://{host.hostname}/api/graphql` for `ghes`.
- The PAT must not be logged, included in error messages, or copied into the SQLite database.

For GitHub.com, the default host values should be equivalent to:

```yaml
host:
  name: "GitHub.com"
  scheme: "https"
  hostname: "api.github.com"
  rest_api_base_path: "/"
  kind: "github"
```

### FR-8b First-run setup

On first launch, if the YAML configuration file does not yet contain the required host and PAT information, the application must present a setup flow before showing the main UI.

The first-run setup flow must collect at least:

- Host display name
- Host scheme
- Host hostname
- Host REST API base path
- Host kind
- PAT

The first-run setup flow should offer a connection test action.

Connection testing is optional and must not be a prerequisite for entering the main application if the user chooses to continue with existing local data.

## 7. Non-Functional Requirements

### NFR-1 Platform support

The app must be buildable for:

- Linux desktop environments used on Ubuntu
- Windows desktop environments

### NFR-2 Local persistence

State required for daily use must persist locally without requiring a separate service.

The persistence boundary for v1 is:

- Plain text YAML configuration file for application settings and credentials
- Local database for fetched GitHub data, saved queries, and per-item local state such as unread, bookmark, and archive

### NFR-3 Extensibility

The code structure should make it practical to add:

- More authentication methods
- More than one host in a future version
- More stream filters and views
- Additional GitHub search resource types in future versions

### NFR-4 Operational independence

The application should work without any custom backend operated by the project.

### NFR-5 Refresh behavior

The application must support background refresh by polling the configured GitHub host and foreground refresh through an explicit user action.

The default polling interval must be 5 minutes and must be configurable by the user.

### NFR-6 Storage split

v1 must separate configuration storage from content storage.

- The YAML configuration file stores user-editable settings and the PAT in plain text
- The local database stores saved queries, fetched GitHub items, and local item state

## 8. Data Model Draft

The following logical entities are required:

| Entity | Purpose | Notes |
| --- | --- | --- |
| HostConfig | API endpoint and auth target | GitHub.com or GHES |
| AuthCredential | Authentication material for a host | v1 uses PAT |
| SavedQuery | Named GitHub search definition | Bound to a host |
| StreamItem | Materialized item shown in the UI | Backed by GitHub search results |
| ItemState | Local flags for unread/bookmarked/archived | Stored locally |

Storage mapping for v1:

| Storage | Content |
| --- | --- |
| YAML config file | `HostConfig`, `AuthCredential`, polling interval, UI settings, sort preferences |
| Local database | `SavedQuery`, `StreamItem`, `ItemState`, and other cached GitHub-derived data |

### 8a. SQLite Storage Contract

The local database must be SQLite. The default database path should follow platform data-directory conventions:

- Linux: `$XDG_DATA_HOME/ghstreamlistner/ghstreamlistner.db` when `XDG_DATA_HOME` is set, otherwise `~/.local/share/ghstreamlistner/ghstreamlistner.db`
- Windows: `%LOCALAPPDATA%/ghstreamlistner/ghstreamlistner.db`

The database must not store the PAT. Host identity is represented by a stable local `host_id` so saved query and item rows can survive host display-name edits. Since v1 supports one configured host, the application should create one local host row when the database is initialized or when the host fingerprint changes.

The host fingerprint is:

```text
{kind}|{scheme}|{hostname}|{normalized_rest_api_base_path}
```

The schema version must be tracked through `PRAGMA user_version`; v1 schema version is `1`.

Required DDL:

```sql
PRAGMA foreign_keys = ON;

CREATE TABLE hosts (
    id INTEGER PRIMARY KEY,
    fingerprint TEXT NOT NULL UNIQUE,
    name TEXT NOT NULL,
    kind TEXT NOT NULL CHECK (kind IN ('github', 'ghes')),
    scheme TEXT NOT NULL CHECK (scheme IN ('https', 'http')),
    hostname TEXT NOT NULL,
    rest_api_base_path TEXT NOT NULL,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL
);

CREATE TABLE saved_queries (
    id INTEGER PRIMARY KEY,
    host_id INTEGER NOT NULL REFERENCES hosts(id) ON DELETE CASCADE,
    name TEXT NOT NULL,
    query TEXT NOT NULL,
    resource_type TEXT NOT NULL DEFAULT 'issue_or_pull_request'
        CHECK (resource_type IN ('issue_or_pull_request')),
    sort TEXT NOT NULL DEFAULT 'updated_desc'
        CHECK (sort IN (
            'updated_desc',
            'updated_asc',
            'created_desc',
            'created_asc',
            'comments_desc',
            'comments_asc'
        )),
    enabled INTEGER NOT NULL DEFAULT 1 CHECK (enabled IN (0, 1)),
    position INTEGER NOT NULL DEFAULT 0,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,
    last_successful_sync_at TEXT,
    last_sync_error TEXT,
    UNIQUE (host_id, name)
);

CREATE TABLE stream_items (
    id INTEGER PRIMARY KEY,
    host_id INTEGER NOT NULL REFERENCES hosts(id) ON DELETE CASCADE,
    node_id TEXT,
    repository_owner TEXT NOT NULL,
    repository_name TEXT NOT NULL,
    number INTEGER NOT NULL,
    item_type TEXT NOT NULL CHECK (item_type IN ('issue', 'pull_request')),
    title TEXT NOT NULL,
    author_login TEXT,
    html_url TEXT NOT NULL,
    api_url TEXT,
    state TEXT NOT NULL CHECK (state IN ('open', 'closed')),
    is_draft INTEGER CHECK (is_draft IN (0, 1)),
    is_merged INTEGER CHECK (is_merged IN (0, 1)),
    review_status TEXT CHECK (review_status IN (
        'none',
        'review_required',
        'changes_requested',
        'approved',
        'unknown'
    )),
    comment_count INTEGER NOT NULL DEFAULT 0,
    created_at_github TEXT NOT NULL,
    updated_at_github TEXT NOT NULL,
    closed_at_github TEXT,
    merged_at_github TEXT,
    last_seen_at TEXT NOT NULL,
    last_enriched_at TEXT,
    raw_search_json TEXT,
    raw_graphql_json TEXT,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,
    UNIQUE (host_id, repository_owner, repository_name, number, item_type)
);

CREATE TABLE stream_item_labels (
    stream_item_id INTEGER NOT NULL REFERENCES stream_items(id) ON DELETE CASCADE,
    name TEXT NOT NULL,
    color TEXT,
    description TEXT,
    PRIMARY KEY (stream_item_id, name)
);

CREATE TABLE stream_item_assignees (
    stream_item_id INTEGER NOT NULL REFERENCES stream_items(id) ON DELETE CASCADE,
    login TEXT NOT NULL,
    avatar_url TEXT,
    PRIMARY KEY (stream_item_id, login)
);

CREATE TABLE saved_query_matches (
    saved_query_id INTEGER NOT NULL REFERENCES saved_queries(id) ON DELETE CASCADE,
    stream_item_id INTEGER NOT NULL REFERENCES stream_items(id) ON DELETE CASCADE,
    first_seen_at TEXT NOT NULL,
    last_seen_at TEXT NOT NULL,
    search_rank INTEGER,
    PRIMARY KEY (saved_query_id, stream_item_id)
);

CREATE TABLE item_state (
    stream_item_id INTEGER PRIMARY KEY REFERENCES stream_items(id) ON DELETE CASCADE,
    is_unread INTEGER NOT NULL DEFAULT 1 CHECK (is_unread IN (0, 1)),
    is_bookmarked INTEGER NOT NULL DEFAULT 0 CHECK (is_bookmarked IN (0, 1)),
    is_archived INTEGER NOT NULL DEFAULT 0 CHECK (is_archived IN (0, 1)),
    read_at TEXT,
    unread_at TEXT,
    bookmarked_at TEXT,
    archived_at TEXT,
    updated_at TEXT NOT NULL
);

CREATE INDEX idx_saved_queries_host_position
    ON saved_queries(host_id, enabled, position, name);

CREATE INDEX idx_stream_items_host_updated
    ON stream_items(host_id, updated_at_github DESC);

CREATE INDEX idx_stream_items_host_repo_number
    ON stream_items(host_id, repository_owner, repository_name, number);

CREATE INDEX idx_saved_query_matches_item
    ON saved_query_matches(stream_item_id);

CREATE INDEX idx_item_state_flags
    ON item_state(is_archived, is_unread, is_bookmarked);
```

Timestamp columns must store UTC RFC 3339 strings. Boolean columns use integer values `0` and `1`. JSON cache columns are optional implementation aids; rendering must use the normalized columns above and must not require reparsing cached JSON.

When a search result is first inserted into `stream_items`, the app must create an `item_state` row with `is_unread = 1`, `is_bookmarked = 0`, and `is_archived = 0`. Later refreshes must update GitHub-derived metadata without overwriting local state flags.

Saved query refresh behavior:

- REST Search results upsert `stream_items`, `stream_item_labels`, `stream_item_assignees`, and `saved_query_matches`.
- If the same item matches multiple saved queries, it must have one `stream_items` row and multiple `saved_query_matches` rows.
- Refresh must update `saved_query_matches.last_seen_at` for returned items.
- v1 must not delete or hide cached items merely because a later refresh fails.
- v1 may keep stale `saved_query_matches` rows when an item stops appearing in a query, because GitHub Search qualifiers can be state-dependent. The item should leave a query view only when the local DB state and current query/filter rules no longer match that view, or when a successful implementation explicitly reconciles the query result set.
- `last_sync_error` stores a short user-displayable error category or message and must not include secrets.

Required view semantics:

- Saved query unread badge: count distinct `stream_items` joined through `saved_query_matches` where `item_state.is_unread = 1` and `item_state.is_archived = 0`.
- Inbox: all distinct non-archived items across enabled saved queries.
- Bookmark: all distinct bookmarked, non-archived items across enabled saved queries.
- Archived: all distinct archived items across enabled saved queries, including items that are still unread.
- Saved query view: items matched to that query, excluding archived items by default.
- `Open` filter: issues with `state = 'open'`; pull requests with `state = 'open'` and `is_merged = 0`; draft pull requests are included.
- `Unread` filter: `item_state.is_unread = 1`.
- `Bookmarked` filter: `item_state.is_bookmarked = 1`.

Default list ordering is `stream_items.updated_at_github DESC`. Other v1 ordering options must map directly from the `ui.default_sort` and `saved_queries.sort` enum values.

### 8b. GraphQL Enrichment Contract

REST Search is the discovery source. GraphQL enrichment must only run for already discovered items and should batch requests where practical.

For issues and pull requests, the app must persist the following from REST Search before enrichment:

- GitHub node ID if available
- Repository owner and repository name
- Issue or pull request number
- Item type
- Title
- Author login
- HTML URL
- API URL
- State
- Comment count
- Labels
- Assignees
- Created, updated, and closed timestamps

For pull requests, GraphQL enrichment must populate or refresh:

- `isDraft`
- `merged`
- `mergedAt`
- Review decision-derived status
- Review request presence

The v1 GraphQL selection set for pull requests should include:

```graphql
nodes(ids: $ids) {
  ... on PullRequest {
    id
    number
    title
    state
    isDraft
    merged
    mergedAt
    reviewDecision
    reviewRequests(first: 20) {
      totalCount
    }
    latestReviews(first: 20) {
      nodes {
        state
        author {
          login
        }
        submittedAt
      }
    }
  }
}
```

The persisted `review_status` value must be derived as follows:

- `changes_requested` when `reviewDecision` is `CHANGES_REQUESTED` or the latest review set contains an effective changes-requested review.
- `approved` when `reviewDecision` is `APPROVED`.
- `review_required` when review requests exist or `reviewDecision` is `REVIEW_REQUIRED`.
- `none` when no review signal exists for an open pull request.
- `unknown` when GraphQL enrichment failed or the PR metadata is unavailable.

For issues, GraphQL enrichment is optional in v1 unless REST Search does not provide a required display field. Failed enrichment must not prevent REST Search data from being stored or rendered.

### 8c. Rust Module Boundaries

The initial implementation should use these module boundaries:

| Module | Responsibility |
| --- | --- |
| `src/main.rs` | Native startup, top-level dependency construction, and `eframe::run_native` only |
| `src/app/mod.rs` | `egui` application state, screen routing, refresh triggers, and UI-to-domain command dispatch |
| `src/app/setup.rs` | First-run setup flow and config validation display |
| `src/app/stream.rs` | Two-pane stream UI, list rendering, filters, item actions, and status presentation |
| `src/config.rs` | YAML file path resolution, parse/write, validation, normalization, and PAT redaction helpers |
| `src/models.rs` | Shared domain types and enums for config, queries, items, filters, sort orders, and local state |
| `src/storage/mod.rs` | SQLite connection management, migrations, transactions, and repository traits |
| `src/storage/schema.rs` | Schema version constants and migration SQL |
| `src/storage/queries.rs` | Saved query persistence and list/read-count queries |
| `src/storage/items.rs` | Stream item upsert, match tracking, metadata persistence, and local item state updates |
| `src/github/mod.rs` | GitHub service facade used by the app layer |
| `src/github/rest.rs` | REST Search request construction, pagination, response parsing, and rate/error mapping |
| `src/github/graphql.rs` | GraphQL enrichment request construction, response parsing, and metadata derivation |
| `src/sync.rs` | Manual refresh and polling orchestration; DB-backed write-before-render flow |

`anyhow` may be used in `src/main.rs` and other binary startup code only. Library-style modules should define typed errors with `thiserror`.

The app layer must not render directly from GitHub response structs. Network responses flow through `github`, are merged by `sync`, are written through `storage`, and are then read back from `storage` for UI rendering.

## 9. Error Handling Expectations

- Invalid host configuration must surface a user-visible validation error.
- Authentication failures must be visible to the user and tied to the relevant host.
- Query execution failures must not silently clear locally stored item state.
- Local database failures must be surfaced explicitly.
- Configuration file parse and validation failures must be surfaced explicitly.
- The application must clearly communicate that the PAT is stored in plain text configuration in v1.
- Authentication, network, and API failures should preserve the last successfully stored list in the right pane.
- Communication and sync status should be exposed through a lightweight status indicator instead of replacing the main list content.

## 10. Acceptance Criteria Draft

### AC-1 Host and PAT setup

**Given** a user opens the app for the first time  
**When** they configure a GitHub or GHES host with a PAT  
**Then** the configuration is written to the YAML settings file, including the PAT, and can be reused after restart

### AC-2 Saved query lifecycle

**Given** a configured host  
**When** the user creates, edits, or deletes a saved query  
**Then** the updated query list is reflected in the app and persists after restart via the local database

### AC-2a Refresh behavior

**Given** at least one saved query exists  
**When** the user triggers a manual refresh  
**Then** the app requests updated results for the relevant query, enriches items with any required additional API data, and updates the stream without requiring an app restart

**Given** at least one saved query exists  
**When** automatic polling is enabled and the polling interval elapses  
**Then** the app refreshes the stream using the configured host and preserves local unread, bookmark, and archive state

**Given** automatic polling is enabled  
**When** the user changes the polling interval  
**Then** the new interval is persisted in the YAML settings file and used for subsequent automatic refreshes

### AC-2b Ordering behavior

**Given** the stream is displayed with default settings  
**When** the user first opens the app

## 12. Risks and Deferred Decisions

- PAT is intentionally stored in plain text in the YAML configuration file in v1, which is a known security trade-off.
- GitHub Search polling has known state-transition limitations for qualifiers such as open/closed, merged, and draft.

## 13. Initial Implementation Handoff

Likely implementation areas:

- `src/main.rs`: bootstrap entry point
- `src/app/`: application state and screen routing
- `src/github/`: host configuration, API client, query execution
- `src/storage/`: local database and persisted item state
- `src/models/`: host/query/item domain models

The current repository appears to contain only a minimal `Hello world` scaffold, so the implementation will require establishing the application structure in addition to feature code.
